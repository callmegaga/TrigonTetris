# 功能开发生命周期

## 阶段 1：需求进入

输入可以是业务描述、原型、设计稿、接口说明或缺陷单。

若需求缺少关键实现决策，必须先澄清，再进入规格编写。典型澄清项包括但不限于：

- 包管理工具，如 `npm`、`pnpm`
- 核心框架和版本，如 Vue、Vite、TypeScript
- 是否接入特定依赖，如 Element Plus、Pinia、Vue Router、ECharts
- 是否需要测试框架、Lint、Prettier、CI、mock、harness
- 是否有既定目录结构、代码风格、部署方式或运行环境约束
- 是否有设计稿、接口契约、权限模型或埋点要求

若以上信息不明确，AI 不得自行假设后直接初始化或大规模实现，必须先向需求方逐项确认关键决策。

若需求涉及项目初始化、依赖安装、依赖升级、脚手架生成、工具链迁移等工作，AI 必须先查询最新官方文档，再给出推荐方案。默认优先采用官方当前推荐的安装方式、版本范围和配置方案。

若当前 AI 工具支持 `superpowers`，则在本阶段进行需求梳理、范围识别、风险拆分和逻辑澄清时，应优先使用 `superpowers` 辅助完成。

出口条件：

- 已明确目标与范围
- 已明确关键技术决策与约束
- 已创建对应功能目录 `specs/<功能名>/`
- 已填写 `proposal.md`

## 阶段 2：规格澄清

补齐功能行为、状态、接口、边界条件和验收标准。

若当前 AI 工具支持 `superpowers`，则在生成 `proposal`、`spec`、`tasks` 前，应优先使用 `superpowers` 进行需求逻辑梳理，确保规格结构清晰、依赖关系明确、风险点可追踪。

若本阶段涉及依赖或工具选型，规格中应记录：

- 官方文档来源
- 采用的版本依据
- 采用该方案而非备选方案的原因

出口条件：

- `spec.md` 达到可执行程度
- 如有设计稿，已产出 UI 分析清单
- 风险和未决问题已记录

## 阶段 3：任务拆解

将实现工作切分为可独立验证的任务。

在 `proposal/spec/tasks` 以及必要的测试方案、harness 方案产出后，必须主动向需求方确认这些内容是否满足预期。未获得明确认可前，不得进入实现阶段。

若任务将由多人或多 AI agent 并行处理，则必须遵守 `docs/process/multi-agent-collaboration-contract.md`，并在任务拆解时明确 owner、write scope、read dependencies 和 integrator。

若存在安装、升级或初始化方案，还必须同步确认：

- 是否接受当前基于官方文档的推荐版本
- 是否接受当前包管理工具和依赖选型
- 是否允许执行安装或升级命令

出口条件：

- `tasks.md` 已完成
- 每个任务都能映射到 `spec`
- 明确测试与 harness 要求
- 多 agent 场景下已声明 owner 和 write scope
- 已获得需求方对规格与测试方案的明确确认

## 阶段 4：实现

按任务执行代码开发，不允许脱离 `spec` 私自扩展范围。

出口条件：

- 代码实现完成
- 同步更新受影响文档
- 类型检查、lint、基础测试通过

## 阶段 5：验证

使用测试和 harness 验证实现与 `spec` 一致。

出口条件：

- `acceptance.md` 已填写
- 关键流程通过
- UI 和异常状态经过验证

## 阶段 6：收尾

沉淀变更结论，准备后续迭代和上线。

出口条件：

- 已记录遗留问题
- 必要的截图、日志、测试记录可追踪
- 文档状态更新为完成
