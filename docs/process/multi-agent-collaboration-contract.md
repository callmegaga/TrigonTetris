# 多 Agent 协作契约

## 目标

本契约用于约束多人协作、多 AI agent 协作时的职责边界、写入范围、交接方式和冲突处理，降低长期维护中的覆盖、漂移和责任不清。

## 适用场景

- 一个需求拆成多个并行子任务
- 多个 AI agent 分别处理不同模块
- 需要人工与 AI 混合协作

## 核心原则

- 每个任务必须有明确 owner
- 每个任务必须声明写入范围
- 未声明写入范围，不得并行开发
- 共享文件默认由集成人负责合并
- 交接必须提供可复核产物，而不是口头说明

## 角色定义

### 需求 owner

- 负责确认需求范围、优先级和最终取舍

### spec owner

- 负责维护 `proposal/spec/tasks/acceptance`
- 负责确保规格与实现一致

### 执行 owner

- 负责具体子任务实现
- 只能修改自己声明的写入范围

### verifier

- 负责校验测试、harness、UI 验收或回归结果

### integrator

- 负责合并共享文件改动
- 负责处理多个 agent 的冲突和最终收口

## 任务最小字段

每个可并行任务至少应包含：

- `任务 ID`
- `owner`
- `write scope`
- `read dependencies`
- `输出物`
- `验证命令`
- `交接说明`

## 写入范围规则

- 写入范围应尽量是独立目录、独立模块或独立文件集
- 两个任务不得默认共享同一写入范围
- 若必须改同一文件，必须指定唯一集成人
- `package.json`、全局路由、全局状态、CI、共享规范文档默认视为共享文件

## 交接要求

任务完成后必须提交以下信息：

- 实际修改的文件列表
- 执行过的命令
- 是否通过验证
- 剩余风险
- 需要集成人处理的事项

## 冲突处理

- 若两个 agent 需要修改同一共享文件，必须先升级为集成任务
- 若实现与 spec 冲突，优先回到 `spec owner`
- 若不同 agent 对方案理解不一致，先停实现，再澄清规格

## 与 SDD / Harness 的关系

- SDD 负责定义“要做什么”
- Harness 负责定义“怎么证明做对了”
- 多 agent 契约负责定义“谁改什么、怎么交接、谁来收口”

三者必须同时存在，才能支撑长期维护。
