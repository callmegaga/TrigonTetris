# Harness 说明

本目录用于放置“可重复执行的验证装置”，用于证明实现符合 `spec`，而不是只依赖人工口头确认。

## Harness 与普通测试的区别

- 普通测试更关注函数、模块或组件的正确性
- Harness 更关注一个需求是否被完整交付，强调输入、场景、证据和回归

## 建议结构

- `fixtures/`：固定输入输出数据
- `ui/`：页面截图回归、关键状态校验
- `api/`：接口 mock、契约样例、错误注入
- `flows/`：关键业务流程脚本

## 使用原则

- 每个重要功能至少对应一种 harness
- harness 必须能映射回 `spec`
- harness 失败时应能定位到场景、输入和期望
