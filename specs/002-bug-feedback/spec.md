# Bug Feedback Spec

## Background

当前项目是一个基于 Vue 3 + Vite 的前端游戏，线上部署后需要一个低门槛的 BUG 反馈入口，帮助玩家在问题发生时直接提交现场信息。仅依靠文字描述很难稳定复现问题，因此反馈能力需要覆盖“文字描述 + 现场快照 + 后台查看 + 现场还原”这一完整链路。

当前版本的目标是快速还原玩家提交反馈当刻的游戏现场，用于人工排查与修复，不要求从该现场继续运行后仍与玩家原始会话完全一致。

## Goal

- 为玩家提供一个随时可用的 BUG 反馈入口
- 在反馈提交时自动采集当前游戏现场
- 将反馈内容持久化存储到 Vercel 可用的免费能力范围内
- 为维护者提供一个轻量后台，查看反馈列表、查看详情、恢复现场
- 支持将某条反馈对应的游戏现场重新渲染出来

## Non-Goals

- 不要求记录完整输入历史
- 不要求保证恢复现场后的后续随机演化与原始会话一致
- 不要求在当前版本实现完整工单系统、评论系统或多人协作流
- 不要求在当前版本实现复杂权限体系
- 不要求在当前版本支持模糊搜索、全文检索或高级统计分析

## Terms

- `Feedback Button`: 游戏左上角的 BUG 反馈入口
- `Feedback Modal`: 玩家填写 BUG 描述的弹窗
- `Game Snapshot`: 用于恢复游戏当刻现场的结构化快照
- `Feedback Report`: 一条完整反馈记录，包含描述、截图、快照和元信息
- `Restore View`: 用于根据 `Game Snapshot` 重新渲染现场的调试页面或调试模式
- `Admin View`: 查看反馈列表和反馈详情的后台页面

## Scope

- 玩家侧：反馈入口、弹窗、现场采集、提交反馈
- 服务端：接收反馈、校验数据、持久化存储
- 维护侧：反馈列表、反馈详情、恢复现场

## Player Rules

1. 游戏左上角必须提供固定可见的 `BUG反馈` 按钮。
2. `BUG反馈` 按钮在欢迎页、游戏进行中和游戏结束后都应可使用。
3. 玩家点击 `BUG反馈` 按钮后，当前游戏应冻结在提交当刻的画面与状态。
4. 游戏冻结后，应弹出 `Feedback Modal` 供玩家填写描述。
5. 描述输入框内容上限为 100 字。
6. 界面应向玩家显示剩余字数或当前字数。
7. 玩家可以取消反馈；取消后游戏应退出反馈流程。
8. 若反馈提交成功，界面应明确提示成功结果。
9. 若反馈提交失败，界面应明确提示失败结果，且不得让玩家误以为已经提交成功。
10. 在提交请求未完成前，应避免重复提交同一条反馈。

## Capture Rules

1. 每条反馈必须包含一份 `Game Snapshot`。
2. 每条反馈应尽量包含一张提交当刻的游戏截图。
3. `Game Snapshot` 必须足以恢复提交当刻的主要游戏现场。
4. `Game Snapshot` 至少应包含：
   - 游戏状态
   - 当前分数
   - 历史最高分
   - 棋盘数据
   - 当前活动块信息，若存在
   - 后续预览块信息
5. `Game Snapshot` 中的棋盘数据应使用可序列化的轻量结构，不得依赖运行时对象引用。
6. 反馈记录应包含提交时间与基础运行环境信息，以帮助维护者判断问题背景。
7. 当前版本只要求恢复“提交当刻的现场”，不要求记录完整重放信息。

## Validation Rules

1. 描述长度限制必须同时在客户端和服务端校验。
2. 服务端必须拒绝缺少 `Game Snapshot` 的提交。
3. 服务端必须拒绝结构非法或字段缺失的快照。
4. 当截图缺失但快照完整时，服务端仍可接受该反馈。
5. 服务端写入失败时，不得产生“提交成功”的用户反馈。

## Storage Rules

1. 反馈记录必须被持久化存储，而不是只保存在浏览器本地。
2. 当前版本的持久化方案以 Vercel Blob 为基线能力。
3. 每条反馈至少应持久化一份结构化报告文件。
4. 若存在截图，应与结构化报告分开存储。
5. 结构化报告必须包含一个稳定的反馈标识，用于后台详情查看与现场恢复。
6. 结构化报告必须保存该条反馈的处理状态。
7. 当前版本的处理状态至少包括：
   - `new`
   - `processing`
   - `resolved`

## Admin Rules

1. 系统应提供 `Admin View`，用于查看已提交的反馈。
2. `Admin View` 至少应支持反馈列表和反馈详情两个层级。
3. 反馈列表至少应展示：
   - 反馈标识
   - 提交时间
   - 处理状态
   - 描述摘要
   - 关键游戏信息摘要
4. 反馈详情至少应展示：
   - 完整描述
   - 截图，若存在
   - 完整 `Game Snapshot`
   - 处理状态
   - 恢复现场入口
5. 维护者应能在详情页修改反馈处理状态。
6. 当前版本不要求在后台实现复杂筛选、搜索或批量操作。

## Restore Rules

1. 系统必须支持根据某条反馈记录中的 `Game Snapshot` 恢复游戏现场。
2. 恢复现场后的画面应与提交当刻的主要游戏内容一致。
3. 恢复现场至少应覆盖：
   - 主棋盘
   - 当前活动块
   - 后续预览块
   - 分数信息
   - 游戏状态
4. 当前版本允许将恢复现场用于只读调试视图，也允许恢复到可交互实例中。
5. 当前版本不要求恢复后继续运行的结果与原始会话完全一致。

## Data Model Expectations

1. `Game Snapshot` 应显式包含版本号，以支持后续结构演进。
2. `Game Snapshot` 的字段命名应稳定，避免依赖临时调试字段。
3. 新版本若调整快照结构，应保证后台能够识别版本差异。
4. `Feedback Report` 应同时保存玩家描述、快照、截图引用和元信息。

## Acceptance Scenarios

### Scenario 1: Submit feedback during active gameplay

- Given: 玩家正在游戏中，左上角可见 `BUG反馈` 按钮
- When: 玩家点击按钮并填写 100 字以内描述后提交
- Then: 当前游戏现场被冻结并被采集，反馈被持久化存储，界面提示提交成功

### Scenario 2: Submit feedback without exact replay support

- Given: 当前版本未记录完整输入历史与随机数状态
- When: 维护者查看该反馈并执行恢复现场
- Then: 系统应能恢复提交当刻的主要游戏画面，但不要求后续演化与玩家原始会话一致

### Scenario 3: Reject over-limit description

- Given: 玩家输入超过 100 字的描述
- When: 玩家尝试提交
- Then: 界面或服务端必须阻止该提交，并告知描述超出限制

### Scenario 4: Restore scene from admin detail

- Given: 后台存在一条包含合法 `Game Snapshot` 的反馈记录
- When: 维护者在详情页点击恢复现场
- Then: 系统应打开或进入 `Restore View`，并显示该反馈提交当刻的游戏现场

### Scenario 5: Feedback without screenshot

- Given: 某次反馈未成功生成截图，但快照完整
- When: 服务端接收该反馈
- Then: 该反馈仍可被保存，并可在后台查看与恢复现场

### Scenario 6: Storage failure on submit

- Given: 玩家填写了合法描述且快照采集成功
- When: 服务端持久化失败
- Then: 玩家必须收到失败提示，且该反馈不得被标记为已提交成功

## Verification

- `pnpm build`
- 涉及快照结构与恢复逻辑时补充 `Vitest`
- 必要时使用恢复现场页面进行手工可视化验证

## Open Questions

- 后台访问控制采用何种最小可用方案，需要在实现前单独确认
- 恢复现场页面是否允许继续操作游戏，还是只提供只读调试视图，需要在 UI 设计阶段定稿
- 后续若反馈量增长，是否需要引入数据库存储元信息与检索索引，需要在下一阶段评估
