# Spec 评审清单

评审 `spec` 时，至少检查以下内容。

## 完整性

- [ ] 是否说明了背景、目标、非目标
- [ ] 是否定义了主流程、异常流程、边界条件
- [ ] 是否列出了页面状态和错误处理
- [ ] 是否明确了接口、字段和 mock 策略
- [ ] 是否明确了关键技术决策，如包管理工具、核心依赖、版本和目录约束
- [ ] 若涉及初始化或依赖变更，是否给出了官方文档来源和版本依据

## 可执行性

- [ ] 开发人员是否可以直接据此实现
- [ ] 测试人员是否可以直接据此验收
- [ ] 是否存在模糊表述，如“适当”“优化一下”“类似之前”
- [ ] 是否已经补齐会影响实现路线的关键空白，而不是由开发阶段临时决定

## 一致性

- [ ] 是否与 proposal 一致
- [ ] 是否与现有架构和规范一致
- [ ] 是否与设计稿、接口契约一致
- [ ] 安装、升级和初始化方案是否与官方当前推荐方案一致，若不一致是否说明原因

## 可验证性

- [ ] 每条关键行为是否能设计测试或 harness
- [ ] 是否明确了成功标准
- [ ] 是否定义了需要沉淀的证据，如截图、日志、fixture
- [ ] 是否已经产出测试方案或 harness 方案，并等待需求方确认
- [ ] 若涉及安装或升级，是否已经等待需求方确认具体版本与执行方案

## 风险

- [ ] 是否记录技术风险和依赖
- [ ] 是否标出未决问题
- [ ] 是否说明哪些内容本次不做
- [ ] 是否存在尚未确认却会影响初始化或架构选型的决策
- [ ] 若 `Context7` 不可用，是否已说明替代的官方文档查询方式
